# План офлайн-пилота LLM-анализатора (без real-time)

**Цель:** проверить гипотезу, что LLM улучшает классификацию сообщений (сигнал / шум, актив, важность), **не добавляя** постоянный Telethon real-time loop и не блокируя API.

**Связано:** `docs/ADR_RT_TELETHON_LLM_EVALUATION.md`, `docs/DATA_PLANE_MIGRATION.md`, `docs/PIPELINE_INGESTION_AND_PARSING.md`.

## Объём пилота

| Параметр | Значение |
|----------|----------|
| Режим | **Офлайн**: батч по историческим сообщениям или экспорту из `raw_events` / legacy таблиц. |
| Частота | По запросу или cron (например, раз в неделю на фиксированной выборке). |
| Объём | Начать с 500–2000 сообщений (стратификация: явные сигналы, шум, пограничные). |

## Входные данные

- Идентификатор сообщения (source, external_id или hash текста + timestamp).
- Сырой текст (и при наличии — метаданные канала).
- Опционально: результат legacy-парсера для сравнения (не как истина, а как baseline).

Источник строк: read-only выборка из БД или экспорт CSV/JSON в `temp/` (без секретов в репозитории).

## Выходные данные

- Таблица **`llm_message_analysis`** (или префикс `pilot_`) с полями, например:

  - `message_ref` (FK или составной ключ)
  - `model_id`, `prompt_version`
  - `is_signal` (bool / score)
  - `assets[]`, `direction`, `confidence`
  - `labels`: sentiment, importance, scam_risk (по согласованной схеме)
  - `raw_llm_json` (для аудита)
  - `created_at`

- Либо файлы JSONL в `temp/` для первого прогона, с последующей загрузкой в БД.

## Процесс

1. **Скрипт** (например, `scripts/offline_llm_pilot_v1.py` или задача Celery с низким приоритетом): читает батч → вызывает LLM (HTTP к Ollama/OpenAI) с таймаутом и ретраями.
2. **Идемпотентность:** повторный прогон по тому же `message_ref` + `prompt_version` перезаписывает или пропускает (явная политика).
3. **Сбои LLM:** не влияют на основной API; логирование и счётчик `llm_failures`.

## Метрики успеха

| Метрика | Как считать |
|---------|-------------|
| Согласие с золотой разметкой | Precision/recall по сигнал/не-сигнал (если есть разметка). |
| Полезность для отсева шума | Доля сообщений, помеченных как low-importance, совпадающих с экспертной оценкой. |
| Стабильность | Два прогона с temperature=0 дают одинаковый JSON (или допустимый diff). |
| Стоимость / время | Токены и секунды на 1k сообщений. |

**Критерий «идём в real-time»:** только если метрики выше **и** базовый data plane стабилен (см. ADR); иначе пилот остаётся исследовательским.

## Эволюция после пилота (из плана продукта)

1. **Шаг 1:** офлайн-батч по историческим сообщениям → сохранение JSON-анализов (таблица или JSONL).
2. **Шаг 2:** проверка гипотез — шум, выборка для ML, прирост к метрикам vs парсер.
3. **Шаг 3:** real-time listener **только** если шаги 1–2 убедительны и базовый пайплайн стабилен; иначе не поднимать `events.NewMessage` + LLM в прод.

## Вне скоупа пилота

- `events.NewMessage` в реальном времени.
- Запись в критичные для API таблицы без feature-флага.
- Обязательная LLM-обработка в пользовательском пути.

## Следующие шаги после пилота

1. Отчёт: таблица метрик + примеры ошибок LLM vs парсер.
2. Решение: продолжить офлайн / расширить корпус / отложить до закрытия приёмки data plane.
