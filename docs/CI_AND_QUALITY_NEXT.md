# CI и качество: следующие шаги (приближение к production)

**Дата:** 2026-04-03  

Практические шаги из ревью; выполнять по мере готовности, чтобы не ломать `main`.

## 1. Покрытие тестами

- **Факт:** в `.github/workflows/ci.yml` для backend задано `--cov-fail-under=30` (цель ТЗ — выше).
- **План:** стабильно поднимать порог (например 35 → 45 → 60) только после того, как локально `pytest --cov=app` стабильно выше нового значения.
- **Документ:** актуальный порог всегда дублировать в `docs/TZ_GAPS_REMEDIATION.md`.

## 2. E2E как gate

- **Факт:** job `e2e` с `continue-on-error: true` не блокирует регрессии.
- **План:** стабилизировать Playwright (тайминги, фикстуры), затем убрать `continue-on-error` для ветки `main` или вынести E2E в отдельный обязательный workflow.

## 3. Нагрузочное тестирование как артефакт

- Прогнать `scripts/load-test/k6-load.js` / `k6-stress.js` на поднятом стеке.
- Зафиксировать RPS, p95, error rate в `docs/LOAD_TEST_RESULTS.md` (см. шаблон таблицы там).

## 4. Контракт API (опционально)

- OpenAPI уже генерируется FastAPI; при росте команды — schemathesis / Dredd против `/openapi.json` в CI.

## 5. Безопасность

- Следовать `docs/SECURITY_AUDIT_RUNBOOK.md` (pre-commit, при необходимости TruffleHog по истории).
