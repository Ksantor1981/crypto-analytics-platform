# ADR: оценка real-time Telethon + LLM (не внедрять преждевременно)

**Статус:** принято (архитектурная позиция)  
**Дата:** 2026-04-04  
**Связано:** `docs/DATA_PLANE_MIGRATION.md`, `SPEC.md`, `docs/TZ_APPENDIX_DATA_PLANE_ACCEPTANCE.md`, `docs/PIPELINE_INGESTION_AND_PARSING.md`, `docs/OFFLINE_LLM_PILOT_PLAN.md`

## Контекст

Предлагается добавить **постоянный real-time listener** (Telethon `events.NewMessage`) и **LLM** для глубокой интерпретации (is_signal, asset, sentiment, importance, scam/шум), с сохранением LLM-анализов и уведомлениями.

Параллельно в продукте уже идёт **канонический data plane** (raw → extraction → normalized → outcomes) и **legacy** контур (`Signal`, polling/scraper).

## Критическая оценка

### Плюсы (долгосрочно)

| Аспект | Эффект |
|--------|--------|
| Product / UX | Real-time + «умный» слой усиливает ценность (алерты по важным событиям, фильтрация шума). |
| Данные | Семантические признаки (importance, риски, narrative) полезны для ML и аналитики каналов. |
| Форматы | LLM лучше покрывает нестандартные сообщения, где regex/OCR слабы. |

### Минусы и риски (сейчас)

| Риск | Почему критично |
|------|-----------------|
| Архитектура | Ещё один долгоживущий процесс (Telethon loop + LLM + уведомления) поверх не до конца стабилизированного контура. |
| Внешняя зависимость | Латентность, стоимость, мониторинг, новые отказы (Ollama/облако). |
| Фокус | Текущие боли: **надёжный сбор**, **аккумуляция**, **базовое распознавание**, приёмка data plane. Real-time+LLM **не устраняет** корневые проблемы парсера/OCR/дедупа. |
| Отладка | Непредсказуемость: гонки, качество промптов, лимиты Telegram, сложнее тестировать локально. |

## Влияние на «100% по ТЗ» (`SPEC.md`)

| Критерий ТЗ (типичный) | Влияние real-time + LLM |
|------------------------|-------------------------|
| Нагрузка (напр. 1000 RPS, p95) | **Не помогает**; LLM — тяжёлый путь, ухудшает хвост латентности при синхронной обработке. |
| Покрытие тестами / CI | **Усложняет** — нужны моки LLM, интеграции Telethon, флаги degraded mode. |
| ML accuracy / честная статистика | LLM может дать **признаки**, но только при **стабильном сырье** и золотой разметке; иначе шум в разметке. |
| Качество базового распознавания | **Не заменяет** детерминированный слой; может маскировать ошибки, если нет сравнения с legacy/экстрактором. |
| Приёмка data plane (`TZ_APPENDIX_*`) | Новый слой **не в чек-листе**; отвлекает ресурсы от закрытия A–I. |

**Вывод:** идея **продуктово перспективна**, но **в текущей фазе** внедрение real-time Telethon + LLM **уводит фокус** от закрытия канонического контура и приёмки по ТЗ. Без устойчивого пайплайна «источник → raw → сигнал/экстракция → метрики» LLM остаётся **дорогой надстройкой**, а не фундаментальным исправлением.

## Рекомендуемое направление (вместо немедленного real-time+LLM)

1. **Стабилизировать базовый data pipeline** — единый путь от источников до таблиц сигналов и метрик; без потерь сообщений, предсказуемый дедуп, накопление вместо перезаписи; минимальные метрики качества (объём, доля распознанных, доля мусора).
2. **Укрепить распознавание без LLM** — детерминированный слой (regex/таблицы) + fallback (OCR); корпус реальных сообщений с золотой разметкой; неуверенные кейсы → `uncertain`, без использования в метриках/ML.
3. **Накопительное хранение и пересчёт** — явная схема raw → сигналы → результаты; идемпотентные batch-задачи; экспорт выборок для офлайн-анализа.
4. **LLM только офлайн** — см. `docs/OFFLINE_LLM_PILOT_PLAN.md`; real-time listener — только после стабилизации и убедительного офлайн-пилота.
5. **Цель будущего real-time-слоя** — уведомления / enrichment поверх стабильных сигналов и **тех же контрактов данных**, не третий параллельный способ хранения.

## Архитектурные ограничения (если real-time+LLM вернётся позже)

- Listener **не пишет произвольно в БД** — сообщения в очередь, дальше существующие сервисы и модели.
- LLM — **асинхронно и необязательно** (degraded mode без LLM не ломает продукт) **или** отдельный сервис с SLA и таймаутами.
- Таблицы LLM-анализа — **вспомогательные**, не критичный путь основных API.

## Решение (архитектурное)

1. **Не внедрять** production real-time Telethon + LLM **до** стабилизации базового data pipeline и прогресса по `TZ_APPENDIX_DATA_PLANE_ACCEPTANCE.md` (блоки A–I), кроме явного ADR-исключения.
2. Разрешить **офлайн-пилот** LLM на исторических сообщениях по плану `docs/OFFLINE_LLM_PILOT_PLAN.md` (отдельная таблица/файлы, не критичный путь API).
3. Будущий real-time слой (если будет) — **notification / enrichment** поверх **тех же контрактов**, что канонический контур; без третьего параллельного «источника правды».

## Roadmap «что делать прямо сейчас» (высокий уровень)

| Шаг | Содержание |
|-----|------------|
| 1 | **Диагноз пайплайна:** где слабы OCR, regex, маппинг asset, валидация цен (логи, тесты); зафиксировать 2–3 метрики качества на неделю. |
| 2 | **Локальные улучшения без LLM:** болезненные форматы парсинга; unit/integration-тесты на них и на сценарии сбора. |
| 3 | **Накопление и пересчёт:** данные не теряются и не затираются; документация схемы таблиц и задач пересчёта. |
| 4 | **Офлайн-пилот LLM** по `OFFLINE_LLM_PILOT_PLAN.md`. |
| 5 | **Решение по real-time:** если шаги 1–4 и качество данных приемлемы, а пилот полезен — проектировать облегчённый listener; иначе заморозить real-time+LLM и укреплять ядро. |
