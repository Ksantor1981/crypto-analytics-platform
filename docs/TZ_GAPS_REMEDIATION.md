# Как закрыть разрывы с ТЗ (практическое руководство)

Документ отвечает на вопрос: *что сделать, чтобы проблемы соответствия ТЗ устранить по максимуму.*

---

## 1. ML точность ≥ 87.2% (ТЗ) vs реальные ~41–67%

**Нельзя** «докрутить цифру» в коде без данных.

| Действие | Где | Эффект |
|----------|-----|--------|
| Собрать 1000+ размеченных сигналов (TP/SL resolved) | пайплайн сбора + БД | основа для честной метрики |
| Feature engineering (волатильность, время суток) | `ml-service/` | см. `docs/ML_ACCURACY_DISCREPANCY.md` план D2 |
| Backtest до выката | скрипты / отчёты | снижение сюрпризов |
| A/B версий модели | feature flag + две `.pkl` | ТЗ: сравнение моделей |

Подробно: [`ML_ACCURACY_DISCREPANCY.md`](./ML_ACCURACY_DISCREPANCY.md).

---

## 2. Покрытие тестами > 80% (ТЗ)

Сейчас в CI: `--cov-fail-under=30` (см. `.github/workflows/ci.yml`). Цель ТЗ 80% — **нарастить постепенно**; реалистичные промежуточные пороги (например 45 → 60 → 65) вводить после стабильного роста локального coverage.

| Действие | Как |
|----------|-----|
| Добавлять тесты к модулям с высоким `Miss` в `pytest --cov=app` | `backend/tests/test_*.py` |
| После стабильного роста | поднять `cov-fail-under` в `.github/workflows/ci.yml` (61 → 65 → … → 80) |
| Не ломать CI | повышать порог только когда локально `pytest --cov=app` стабильно выше нового порога |

## 3. 1000 RPS и p95 < 200 ms

| Действие | Где |
|----------|-----|
| Прогон k6 на **поднятом** API | [`scripts/load-test/README.md`](../scripts/load-test/README.md) |
| Зафиксировать RPS / p95 | внести в отчёт или `docs/PLAN_100_TZ_2026.md` |
| Узкие места | профилирование, кэш Redis, индексы БД (частично уже в проекте) |

Без реального стенда критерий ТЗ **не подтверждается** — только подготовка скриптов и процедуры.

---

## 4. Доступность 99.9%

Только **эксплуатация**: VPS/K8s, healthchecks, реплики, алерты. В репозитории — Docker Compose, Prometheus/Grafana, чек-листы деплоя.

---

## 5. Интеграции Binance / Bybit (в тексте ТЗ)

Фактическая валидация цен в продукте опирается на **CoinGecko** (и сопутствующую логику). Прямые коннекторы Binance/Bybit как отдельный пункт ТЗ:

- либо добавить тонкий клиент (REST) с fallback цепочкой: CoinGecko → Binance → …  
- либо **обновить ТЗ** формулировкой «внешние котировки (CoinGecko как primary)» — юридически согласовать с заказчиком.

## 6. Google Analytics

Включение на фронте:

```bash
# frontend/.env.production (или CI secrets)
NEXT_PUBLIC_GA_MEASUREMENT_ID=G-XXXXXXXXXX
```

Компонент: `frontend/src/components/analytics/GoogleAnalytics.tsx` (подключён в `_app.tsx`). Без переменной скрипт не грузится.

---

## 7. Кастомные алерты (Pro) и страница `/alerts`

- **Backend:** модель `CustomAlert`, сервис `custom_alerts_service.py`, RBAC в `rbac_middleware.py` (`/alerts/custom` → Pro).
- **Пробел:** нужен **стабильный REST-router** в `main.py`, связывающий HTTP с `CustomAlertsService`, и при желании UI в профиле/отдельной странице.

Страница `/alerts` сейчас показывает **ленту сигналов из API** и явно помечает, что push-центр Pro — отдельная дорожка.

## 8. Бизнес-KPI ТЗ (пользователи, MRR, NPS)

Не закрываются кодом; нужны маркетинг, продукт и аналитика (GA + внутренние события).
