# Расхождение ML accuracy: ТЗ vs реальность

**См. также:** [TZ_GAPS_REMEDIATION.md](./TZ_GAPS_REMEDIATION.md) — что делать по всем разрывам с ТЗ.  
**Пайплайн данных и честное обучение:** [ML_DATA_INTEGRITY_ROADMAP.md](./ML_DATA_INTEGRITY_ROADMAP.md).

**SPEC (целевой KPI, не текущий факт):** ML ≥87.2% на **подтверждённой** выборке после бэктеста.  
**Факт (исторически):** CV ~96% на синтетике/плейсхолдерах; на OOS и реальных данных — существенно ниже (~41–67% в отдельных аудитах).

---

## Причины расхождения

1. **CV vs реальные данные**
   - Cross-validation на train/test split даёт оптимистичную оценку
   - Реальные сигналы — другой distribution, шум, мусор, дубликаты

2. **Качество данных**
   - Мусорные посты (новости, дайджесты) до фильтрации снижали accuracy
   - После чистки (DEFI_NEWS_LIKE, дедуп) — accuracy ~67.5% (AUDIT_BOEVOJ)

3. **Недостаток объёма**
   - ~150–500 реальных сигналов для обучения
   - Модель переобучается на малой выборке

4. **Feature drift**
   - Фичи из train могут не соответствовать production (цены, волатильность)

---

## План улучшения (D1–D4)

| Этап | Действие | Оценка |
|------|----------|--------|
| D1 | Собрать 1000+ валидированных сигналов (TP/SL resolved) | 2–4 недели |
| D2 | Feature engineering: добавить volatility, volume, time-of-day | 1–2 дня |
| D3 | Backtest на исторических данных до деплоя | 1 день |
| D4 | A/B тест версий модели (feature flag) | 0.5 дня |
