# Парсинг, сбор, хранение, обработка — что есть и как улучшать

Краткий ответ: **блок уже рабочий** (скрап → regex‑парсер → фильтры → БД → валидация → метрики), но он **хрупкий и неоднородный**. Улучшать **можно и нужно** по приоритету ниже.

---

## Как устроено сейчас

| Этап | Где в коде | Заметки |
|------|------------|---------|
| Сбор TG | `telegram_scraper.fetch_channel_posts` | HTML `t.me/s/{user}`, BeautifulSoup |
| Парсинг текста | `parse_signal_from_text` | Regex, словари активов, фильтры news/DeFi/spam |
| Сбор Reddit | `reddit_scraper`, `parsers/reddit_parser.py` | **Не в одном потоке с `run_collection()`** — см. `PIPELINE_SIGNALS.md` |
| Запись | `tasks/collect_signals.run_collection` | Только `platform == "telegram"`; без `entry_price` → отброс |
| Дедуп | `func.left(original_text, 500)` | Быстро, но грубо |
| Постобработка | `historical_validator`, `metrics_calculator` | CoinGecko, accuracy каналов |

Подробная схема: [`PIPELINE_SIGNALS.md`](./PIPELINE_SIGNALS.md).

---

## Сильные стороны

- Явные **анти‑мусор** фильтры (`NEWS_SKIP_KW`, `GARBAGE_SKIP_KW`, `DEFI_NEWS_LIKE` + длина текста).
- **Валидация цен** по диапазонам актива, защита от «$168 million».
- **Санитизация** текста перед сохранением (`sanitizer`).
- **Метрики** пайплайна (`SIGNALS_*` в `metrics.py`).
- Цепочка до **рейтинга каналов** замкнута.

---

## Уязвимости и ограничения

1. **Зависимость от вёрстки Telegram** — смена классов CSS → пустые посты.
2. **Только эвристики** — новые форматы постов («Entry: зона 0.42–0.44», скриншоты без текста) часто не попадут в БД.
3. **Обязательный `entry_price`** — много честных сигналов с «LONG BTC» без явной цены отсекаются.
4. **Reddit / модульный парсер** не в том же цикле, что TG → **разные объёмы и качество** данных.
5. **Дедуп по 500 символам** — и дубли, и ложные уникальные варианты.
6. **Семантика ROI/PnL** между валидатором и метриками — требует выравнивания (уже отмечено в `PIPELINE_SIGNALS.md`).

## Рекомендуемые улучшения (по приоритету)

### P1 — Интеграция и наблюдаемость (низкий риск)

- Включить **Reddit** (и при необходимости `parsers/reddit_parser`) в **один** orchestrator рядом с `run_collection()` или общий `run_all_sources()`.
- Логировать **parse funnel**: `posts_fetched` → `parsed_non_null` → `saved` → `skip_reason` (counters / JSONL).
- Таблица или очередь **`parse_rejects`** (текст, канал, причина) для ручного разбора и обучения правил.

### P2 — Парсер без смены архитектуры

- Расширить **паттерны** entry/TP/SL (зоны `0.42–0.44`, `Entry zone:`, локальные десятичные запятые).
- Поднять **`limit`** постов с 20 для «горячих» каналов (конфиг с БД).
- Нормализовать **хеш дедупа**: `sha256(normalize_whitespace(text))` или fuzzy hash вместо сырого `left(500)`.
- Опционально сохранять сигналы **без entry** со статусом `NEEDS_PRICE` для последующего дообогащения.

### P3 — Качество через данные и ML/LLM

- **Классификатор** «сигнал / не сигнал» (лёгкая модель по TF‑IDF или маленький transformer) *до* тяжёлого regex.
- **LLM‑извлечение** (structured output JSON) для сложных постов — только при превышении порога неопределённости или для Pro‑каналов (стоимость/latency контролировать).
- Разметка **золотого набора** из `parse_rejects` → метрики precision/recall по парсеру.

### P4 — Хранение и обработка

- Заполнять **`message_timestamp`** / `telegram_message_id` из скрапера, если появятся в HTML — лучше для валидатора и дедупа по времени.
- Выровнять поля **PnL % vs absolute** между `historical_validator` и `metrics_calculator` (один источник истины).
- Привести **seed_data** к актуальной модели `Signal`.

### P5 — Надёжность сбора

- Резервный путь: **Telegram Bot API** для каналов, где есть бот (если политика позволяет), если web‑скрап ломается.
- Rate limiting и **ротация User-Agent** / прокси при массовом сборе.

## Итог

| Вопрос | Ответ |
|--------|--------|
| Можно ли улучшить? | **Да**, особенно **единый сбор**, **дедуп**, **расширение правил**, **учёт отказов**, опционально **ML/LLM**. |
| Сейчас «безошибочно»? | **Нет универсально**: regex и HTML хрупки; ошибки частично компенсируются фильтрами и валидатором **после** записи. |
| С чего начать? | **P1 + P2** дают максимум эффекта без тяжёлой инфраструктуры. |
